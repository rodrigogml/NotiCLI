# Project Briefing: NotiCLI

**Data**: 2026-06-26
**Status**: Draft
**Versao**: 1.0

---

## 1. Visao e Proposito

**O que e**: Uma aplicacao CLI nao interativa para outras aplicacoes chamarem por linha de comando e enviarem notificacoes para um usuario.

**Problema que resolve**: Permite que sistemas externos enviem notificacoes por canais como email, Telegram e Slack sem precisarem implementar diretamente cada integracao.

**Proposta de valor**: Oferecer uma chamada simples por CLI, com parametros para usuario, canal, titulo, conteudo, anexos e retorno objetivo por codigo de sucesso ou mensagem de erro.

## 2. Usuarios e Stakeholders

| Ator | Papel | Acoes Principais |
|------|-------|------------------|
| Aplicacao externa | Chamador da CLI | Executa a aplicacao por linha de comando e passa parametros da notificacao |
| Usuario destinatario | Recebedor da notificacao | Recebe notificacoes pelo canal configurado |
| Operador/admin | Configurador | Mantem o arquivo de configuracao com usuarios e credenciais/configuracoes dos canais |

**Stakeholders de decisao**: Usuario do projeto.

## 3. Escopo

### MVP (Essencial)

1. Executar envio por CLI nao interativa.
2. Aceitar parametros de destinatario, canal, titulo, conteudo e anexos.
3. Ler usuarios e configuracoes de canais de um arquivo de configuracao.
4. Suportar canais como email, Telegram e Slack.
5. Retornar sucesso ou erro por exit code e mensagem objetiva, sem interacao.

### Pos-MVP (Desejavel)

Nenhum item programado por enquanto.

### Fora de Escopo

- Interface interativa na CLI.
- Fluxo com perguntas, prompts ou confirmacoes durante a execucao.

## 4. Prioridades e Trade-offs

**Ordem de prioridade**: Qualidade e robustez > CLI clara e previsivel > Escopo completo > Velocidade de entrega.

**Decisoes explicitas**:
- Priorizar robustez mesmo que leve mais tempo.
- Manter a CLI previsivel para ser chamada por outras aplicacoes.
- Aceitar que velocidade de entrega e a menor prioridade relativa, apesar do prazo urgente.

## 5. Restricoes

| Restricao | Valor | Notas |
|-----------|-------|-------|
| Prazo | "Para ontem" | Ha urgencia alta para entrega do MVP |
| Equipe | Usuario + Codex | Equipe minima |
| Budget | Custo zero | Evitar servicos pagos e dependencias com custo operacional |
| Tecnica | Linux agora, Windows portable no futuro | A escolha tecnica deve considerar migracao entre sistemas operacionais |

## 6. Stack Tecnica

| Camada | Tecnologia | Justificativa |
|--------|------------|---------------|
| Linguagem | Go | Binario unico, portavel entre Linux e Windows, adequado para CLI e futuro servico local com API HTTP |
| Frontend | Nenhum | Projeto inicialmente e uma CLI nao interativa |
| Banco de dados | Nenhum | Configuracoes ficam em arquivo |
| Infraestrutura | Execucao local/servidor Linux | A aplicacao sera chamada diretamente por linha de comando |
| Integracoes | Email, Telegram, Slack | Canais de notificacao previstos para o MVP |
| Configuracao | A definir | Usuario conhece melhor JSON, mas prefere o formato mais viavel para Go |

**Alternativa considerada**: Python foi considerado pela velocidade de desenvolvimento, mas Go foi escolhido pela distribuicao como binario unico, portabilidade e melhor adequacao a CLI executada por outras aplicacoes.

## 7. Qualidade e Padroes

**Padroes adotados**:
- Testes rigorosos para parsing da CLI, leitura de configuracao, canais e tratamento de erros.
- Seguranca primeiro para proteger tokens, senhas, arquivos de configuracao, logs e mensagens de erro.
- Observabilidade com logs e diagnosticos uteis para falhas.
- Performance para execucao rapida e baixo overhead por chamada CLI.
- Documentacao com README claro, exemplos de uso e exemplos de configuracao.
- Compatibilidade com Linux no MVP e Windows portable no futuro.

**Compliance**: Nenhum compliance especifico definido.

## 8. Visao de Futuro

**6 meses**: Evoluir para tambem rodar como servico local e aceitar requisicoes por API HTTP.

**12 meses**: Suportar Windows de forma portable e adicionar novos canais de notificacao.

**Riscos conhecidos**:
- Definir mal o formato de configuracao pode dificultar evolucao dos canais.
- Tratamento inadequado de tokens e senhas pode expor credenciais em arquivo, logs ou mensagens de erro.
- Acoplar a arquitetura somente ao modo CLI pode dificultar a evolucao para servico local com API.
